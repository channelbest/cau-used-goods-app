# 前端协作 README

本目录用于 CAU 二手交易平台微信小程序前端开发。当前阶段先完成前端技术方案、页面设计文档和任务拆分，暂不急于写完整业务代码。

## 技术选型

推荐采用 `uni-app + Vue 3` 开发微信小程序端。

- 框架：Vue 3
- 构建/运行：uni-app，编译目标为微信小程序
- 状态管理：Pinia
- 请求封装：统一封装 `uni.request`、`uni.uploadFile`
- 样式：SCSS 或普通 CSS，按项目统一即可
- 调试工具：微信开发者工具、HBuilderX 或 VS Code

与概要设计的差异标注：

- 概要设计中前端层写的是原生微信小程序 `WXML/WXSS/JavaScript/Components`。
- 当前前端建议改为 `uni-app + Vue 3`，最终仍编译运行在微信小程序环境。
- 接口调用能力不变：原设计中的 `wx.request`、`wx.uploadFile` 在 uni-app 中对应为 `uni.request`、`uni.uploadFile`。
- 页面结构、权限规则、核心业务流程仍以 `doc/CAU二手交易平台概要设计报告V2.docx` 为准。

## 页面跳转重点

根据概要设计，前端主流程按以下路径设计：

- 登录认证：开始使用小程序 -> 登录页 -> 学生认证页 -> 首页
- 浏览找货：首页 -> 搜索结果页 / 分类商品页 -> 商品详情页
- 交易预约：商品详情页 -> 提交预约页 -> 订单详情页 -> 评价页
- 举报：商品详情页 / 订单详情页 -> 举报页 -> 举报详情/处理结果页
- 卖家管理：我的页 -> 我的商品页 / 卖家订单管理页 -> 商品编辑页 / 订单处理页
- 消息：首页底部导航消息页 -> 订单详情 / 举报结果 / 公告详情
- 管理员：我的页 -> 管理员入口 -> 用户/商品/举报/敏感词/公告/统计管理

底部导航建议保持 4 个入口：

- 首页
- 发布
- 消息
- 我的

## 分工建议

可以按模块分工：

- 成员 A：登录认证、首页、搜索筛选、商品详情。
- 成员 B：发布商品、商品编辑、图片上传、AI 辅助入口。
- 成员 C：订单、消息、我的页、评价、举报。

也可以按工作类型分工：

- 接口交互 2 人：负责请求封装、接口字段对接、token 处理、错误码处理、上传文件、页面数据联调。
- 页面设计和素材 1 人：负责页面原型、组件样式、图标/图片素材、低保真到高保真整理。

## 前端注意事项

- 商品详情页不展示明文联系方式，只有订单进入 `WAIT_MEET` 后，交易双方可在订单详情页看到必要联系方式。
- 发布、预约、评价、举报等核心操作必须要求已登录且学生认证通过。
- 前端可以控制按钮显示，但不能把它当作权限控制；后端仍必须校验。
- 订单详情页按钮必须根据身份和订单状态动态显示，例如确认、取消、完成、评价。
- 商品图片最多 9 张，图片上传失败时需要明确提示。
- AI 标题优化和 AI 描述生成是扩展功能，失败不能影响手动发布。
- 管理员入口只对 `ADMIN` 角色展示。
- 接口返回统一按 `code/message/data/timestamp/requestId` 结构处理。

## Git 协作命令

本项目采用简化分支模型：

- `main`：稳定分支，由项目组长维护。
- `dev`：日常开发分支，前后端组长审核后合并到这里。
- `feature/*`：个人任务分支，组员在自己的功能分支上开发。

首次拉取：

```bash
git clone <仓库地址>
cd cau-used-goods-app
```

每天开始开发前：

```bash
git checkout dev
git pull origin dev
git status
```

新建个人分支：

```bash
git checkout -b feature/frontend-姓名-模块名
```

提交修改：

```bash
git status
git add frontend
git commit -m "feat(frontend): 完成某某模块设计或页面"
```

推送分支：

```bash
git push origin feature/frontend-姓名-模块名
```

发起 PR/MR：

- 源分支：`feature/frontend-姓名-模块名`
- 目标分支：`dev`
- 审核人：前端组长

合并前同步最新 `dev`：

```bash
git checkout dev
git pull origin dev
git checkout feature/frontend-姓名-模块名
git merge dev
```

冲突处理完成后：

```bash
git add .
git commit -m "chore(frontend): resolve merge conflicts"
git push origin feature/frontend-姓名-模块名
```

## 前端组长审核流程

前端组员完成阶段任务后，不直接合并到 `main`，也不直接推送到 `dev`。组员统一从个人 `feature/*` 分支发起 PR/MR，由前端组长审核前端相关内容，审核通过后合并到 `dev`。

1. 组员新建自己的功能分支：

```bash
git checkout dev
git pull origin dev
git checkout -b feature/frontend-姓名-模块名
```

2. 完成修改后只提交前端相关文件：

```bash
git status
git add frontend
git commit -m "feat(frontend): 完成某某模块"
git push origin feature/frontend-姓名-模块名
```

3. 在代码托管平台发起 Pull Request / Merge Request：

- 目标分支：`dev`。
- 审核人：前端组长。
- 标题建议：`feat(frontend): 某某模块页面/交互/接口处理`

4. PR/MR 描述需写清楚：

- 本次完成的页面或模块。
- 涉及的页面跳转。
- 使用或等待后端提供的接口。
- 与概要设计不一致的地方。
- 仍未完成或需要讨论的问题。

5. 前端组长重点审核：

- 技术实现是否符合 `uni-app + Vue 3` 约定。
- 页面结构和跳转是否符合概要设计。
- P0 核心流程是否走得通。
- 接口封装、错误码处理、token 处理是否统一。
- 登录、认证、权限按钮展示是否符合规则。
- 是否误提交后端、文档之外的无关文件。

6. 审核结果处理：

- 审核通过：由前端组长合并到 `dev`。
- 需要修改：组员在原分支继续提交，PR/MR 会自动更新。
- 有设计分歧：在 PR/MR 评论中标注，并同步到前端设计文档的“差异/问题清单”。

7. 与主分支的关系：

- `dev` 用于前后端日常集成和联调。
- `main` 保持相对稳定，不接收普通组员功能分支直接合并。
- 阶段成果稳定后，由项目组长从 `dev` 合并到 `main`。
