# 税馨乐后端阶段说明

## 1. 当前完成范围

本阶段已完成后端基础启动、数据库连接、统一响应、JWT、认证中间件、`auth` 登录认证模块和 `user` 用户/学生认证模块。

`/auth/wechat-login` 会使用配置中的微信小程序 `app_id` 和 `app_secret` 调用微信 `code2session` 接口，拿到 `openid` 后查询或创建用户，并返回 JWT。

目前前端部分仅用于后端demo调试（真实登录接口中获取code）

## 3. 配置说明

本地运行前运行scripts\sql中脚本，建立数据库，并自己建立数据库管理账号，用于config配置，本地调试

复制配置模板：

```powershell
cd backend
Copy-Item config\config.yaml.example config\config.yaml
```

填写 `backend/config/config.yaml`：

```yaml
server:
  env: dev
  port: 8080

jwt:
  secret: change_this_to_a_long_random_secret
  expire_hours: 168

wechat:
  app_id: your_wechat_mini_program_app_id
  app_secret: your_wechat_mini_program_app_secret

database:
  host: 127.0.0.1
  port: 3306
  user: root
  password: your_mysql_password
  name: cau_used_goods
  charset: utf8mb4
  parse_time: true
  loc: Local
  max_open_conns: 25
  max_idle_conns: 10
  conn_max_lifetime_minutes: 30
```

注意：

- `server.env` 当前支持 `dev` 和 `prod` 两种约定值。
- `dev`：开发模式，会注册 `/auth/dev-login`，方便本地生成普通用户或管理员 token 进行接口测试。
- `prod`：生产模式，不注册 `/auth/dev-login`，只能通过 `/auth/wechat-login` 进行真实微信登录。（即需要使用微信开发者工具，测试号/真实APPID）
- `jwt.secret` 必须替换为较长随机字符串。
- `wechat.app_id` 和 `wechat.app_secret` 必须填写真实微信小程序配置，否则微信登录会失败。
- `database.password` 需要填写本地 MySQL 密码。
- `config.yaml` 包含敏感信息，已加入 `.gitignore`，不要提交。

## 4. 统一响应与错误码

统一响应格式：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "timestamp": "2026-05-24 00:00:00",
  "requestId": "optional-request-id"
}
```

当前采用粗粒度错误码，具体错误原因放在 `message` 中：

| code | 含义 | 常见场景 |
| --- | --- | --- |
| `0` | 成功 | 请求处理成功。 |
| `400` | 请求参数错误 | 请求体格式错误、必填参数缺失、业务参数不合法。 |
| `401` | 未登录或 token 无效 | 缺少 `Authorization`、token 格式错误、token 过期、用户不存在。 |
| `403` | 无权限或状态不允许 | 非管理员访问后台、未完成学生认证访问交易接口。 |
| `404` | 资源不存在 | 预留，后续商品/订单等资源查询可使用。 |
| `409` | 业务冲突 | 预留，后续重复收藏、重复下单等冲突可使用。 |
| `500` | 服务端内部错误 | 数据库错误、微信接口调用失败、服务端异常。 |

示例：

```json
{
  "code": 401,
  "message": "invalid or expired token",
  "data": null,
  "timestamp": "2026-05-24 00:00:00"
}
```

## 5. 接口设计

### 5.1 微信登录

```http
POST /auth/wechat-login
Content-Type: application/json
```

请求体：

```json
{
  "code": "wx_login_code"
}
```

处理流程：

```text
小程序 wx.login 获取 code
-> 前端提交 code
-> 后端调用微信 code2session
-> 获取 openid
-> 查询/创建 users 记录
-> 更新 last_login_time
-> 生成 JWT
```

新用户默认状态：

```text
role = USER
auth_status = UNVERIFIED
account_status = NORMAL
```

响应数据：

```json
{
  "token": "jwt_token",
  "user": {
    "id": 1,
    "nickname": null,
    "avatarUrl": null,
    "role": "USER",
    "authStatus": "UNVERIFIED",
    "accountStatus": "NORMAL"
  }
}
```

*开发登录*

仅 `server.env=dev` 时注册该接口，用于没有前端或没有真实微信 `code` 时进行本地接口测试。`server.env=prod` 时不会注册该接口。

```http
POST /auth/dev-login
Content-Type: application/json
```

普通用户测试：

```json
{
  "openid": "dev_user_openid_001",
  "nickname": "测试用户",
  "role": "USER"
}
```

管理员测试：

```json
{
  "openid": "admin_openid_001",
  "nickname": "管理员",
  "role": "ADMIN"
}
```

响应数据同微信登录，返回 `token` 和 `user`。管理员接口测试时使用管理员登录返回的 token：

```http
Authorization: Bearer <admin_token>
```

### 5.3 修改用户资料

```http
PUT /users/profile
Authorization: Bearer <token>
Content-Type: application/json
```

请求体：

```json
{
  "nickname": "张三",
  "avatarUrl": "https://example.com/avatar.png",
  "phone": "13800000000"
}
```

说明：

- 三个字段均可选，但至少传一个。
- 只更新传入字段。
- `avatarUrl` 可以传微信头像 URL（外链），后端会下载并转存到 `backend/uploads/avatar/`，最终写入 `users.avatar_url` 的是 `/uploads/avatar/文件名` 这种本地访问路径。
- `avatarUrl` 也可以传头像上传接口返回的 `/uploads/avatar/...` 路径，后端会直接保存该路径。
- 学号、真实姓名、学院不在此接口修改，应通过学生认证接口提交。

微信头像转存示例：前端拿到微信头像地址后，调用本接口传入 `avatarUrl`，后端负责下载、保存和更新数据库。

```json
{
  "avatarUrl": "https://thirdwx.qlogo.cn/mmopen/..."
}
```

转存限制：

- 只支持 `http` / `https` 地址。
- 图片大小不超过 2MB。
- 支持 `.jpg`、`.jpeg`、`.png`、`.webp`。

### 5.4 上传/修改用户头像

```http
POST /users/avatar
Authorization: Bearer <token>
Content-Type: multipart/form-data
```

表单字段：

```text

```

限制：

- 文件大小不超过 2MB。
- 支持 `.jpg`、`.jpeg`、`.png`、`.webp`。
- 文件保存到 `backend/uploads/avatar/`。
- 通过 `/uploads/avatar/文件名` 访问。
- 上传成功后会自动写入 `users.avatar_url`。

### 5.5 提交学生认证

```http
POST /users/student-verify
Authorization: Bearer <token>
Content-Type: application/json
```

请求体：

```json
{
  "studentId": "20240001",
  "realName": "张三",
  "college": "信息与电气工程学院"
}
```

当前状态流转：

```text
UNVERIFIED / REJECTED / PENDING -> PENDING
```

说明：

- 提交后不直接变为 `VERIFIED`。
- 后续由管理员模块审核后改为 `VERIFIED` 或 `REJECTED`。
- 如果账号状态不是 `NORMAL`，不允许提交。
- 如果已经是 `VERIFIED`，不允许重复提交。
- 手机号只属于用户资料，不在学生认证接口中提交；如需修改手机号，请调用 `PUT /users/profile`。

### 5.7 查询学生认证审核列表

管理员接口，需要同时满足：

```text
Authorization: Bearer <admin_token>
role = ADMIN
```

```http
GET /admin/users/student-verifications?authStatus=PENDING
Authorization: Bearer <admin_token>
```

说明：

- `authStatus` 可选，默认 `PENDING`。
- 当前支持查询 `PENDING`、`VERIFIED`、`REJECTED`、`UNVERIFIED`。
- 主要用于管理员查看待审核学生认证申请。

响应数据：

```json
{
  "items": [
    {
      "userId": 1,
      "studentId": "20240001",
      "realName": "张三",
      "college": "信息与电气工程学院",
      "authStatus": "PENDING",
      "accountStatus": "NORMAL"
    }
  ]
}
```

### 5.8 审核学生认证

管理员接口，需要同时满足：

```text
Authorization: Bearer <admin_token>
role = ADMIN
```

```http
PUT /admin/users/{id}/student-verify
Authorization: Bearer <admin_token>
Content-Type: application/json
```

审核通过请求体：

```json
{
  "authStatus": "VERIFIED",
  "description": "学生认证审核通过"
}
```

审核驳回请求体：

```json
{
  "authStatus": "REJECTED",
  "description": "学号或姓名信息不匹配"
}
```

当前状态流转：

```text
PENDING -> VERIFIED
PENDING -> REJECTED
```

说明：

- 只能审核 `auth_status=PENDING` 的用户。
- `authStatus` 只允许传 `VERIFIED` 或 `REJECTED`。
- 审核操作会同时写入 `admin_logs`。
- 审核通过写入 `operation_type=STUDENT_VERIFY_APPROVE`。
- 审核驳回写入 `operation_type=STUDENT_VERIFY_REJECT`。

## 6. 中间件设计

### 6.1 `Auth`

文件：`backend/internal/middleware/auth.go`

职责：

- 读取请求头 `Authorization`。
- 校验格式是否为 `Bearer <token>`。
- 解析 JWT。
- 将 `userID` 和 `role` 写入 Gin Context。

### 6.2 `Verified`

文件：`backend/internal/middleware/verified.go`

职责：

- 根据当前 `userID` 查询 `users` 表。
- 校验用户是否满足：

```text
auth_status = VERIFIED
account_status = NORMAL
is_deleted = 0
```

适用接口：

- 发布商品
- 提交预约
- 收藏
- 评价
- 举报
- 其他要求学生认证后才能使用的交易功能

### 6.3 `Admin`

文件：`backend/internal/middleware/admin.go`

职责：

- 校验当前 JWT 中的 `role=ADMIN`。

说明：

- 当前 `Admin` 中间件先基于 JWT 中的 role 判断。
- 后续管理员接口服务层仍应再次查询数据库校验 `role=ADMIN` 和 `account_status=NORMAL`。

## 7. JWT 设计

文件：`backend/pkg/jwt/token.go`

JWT 中只放最小必要信息：

```json
{
  "userId": 1,
  "role": "USER",
  "exp": 1234567890,
  "iat": 1234560000
}
```

设计理由：

- `userId` 用于定位当前用户。
- `role` 用于基础路由权限判断。
- `auth_status` 和 `account_status` 不放入 token，后续由数据库实时查询，避免旧 token 绕过状态变化。
